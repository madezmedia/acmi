# Contributing to ACMI

ACMI is a protocol first and a package second. The most valuable contributions are:

1. **New adapters** for backends we don't cover yet (DynamoDB, Cloudflare KV, FoundationDB, FaunaDB, …).
2. **Reference integrations** for new agent SDKs (Mistral, Cohere, OpenRouter, …).
3. **Bug fixes** to the reference SDK.
4. **Spec clarifications** in `SPEC.md`.

## Setup

```bash
git clone https://github.com/madezmedia/acmi.git
cd acmi
npm install
npm test
```

You should see `31/31 tests passing` from the in-memory conformance suite. If not, file an issue — that's a regression.

## Writing an adapter

The fastest path is to copy `src/adapters/in-memory.ts`, adapt the storage to your backend, and run the conformance suite until it passes.

### Adapter contract

Implement `AcmiAdapter` from `src/types.ts`:

```ts
import type { AcmiAdapter } from "@madezmedia/acmi";

export class MyAdapter implements AcmiAdapter {
  readonly name = "my-backend";

  async profileGet(entityId)     { /* ... */ }
  async profileSet(entityId, doc) { /* ... */ }
  async profileMerge(entityId, partial) { /* ... */ }
  async profileDelete(entityId)  { /* ... */ }

  async signalsGet(entityId, key) { /* ... */ }
  async signalsSet(entityId, key, value) { /* ... */ }
  async signalsAll(entityId) { /* ... */ }
  async signalsDelete(entityId, key) { /* ... */ }

  async timelineAppend(entityId, event) { /* ... */ }
  async timelineRead(entityId, opts) { /* ... */ }
  async timelineSize(entityId) { /* ... */ }

  async close() { /* optional */ }
}
```

### Validate with the conformance suite

```ts
import { runConformanceTests } from "@madezmedia/acmi/testing/conformance";

const result = await runConformanceTests(() => new MyAdapter());
console.log(`${result.passed}/${result.total} passed`);
if (result.failed > 0) {
  for (const r of result.results.filter((r) => !r.pass)) {
    console.error(`  ✗ ${r.name}: ${r.error}`);
  }
  process.exit(1);
}
```

When the suite passes, your adapter is ACMI-conformant. Add it to your repo's README and we'll feature it in the main repo's adapter list.

### Storage tips

- **Profile**: STRING + JSON is simplest. Use a native document type if your backend has one.
- **Signals**: STRING + JSON (single key holds the full map) is what the reference Redis/Upstash adapters use. A native HASH works equally well — both pass conformance.
- **Timeline**: needs ordered range queries by score (= `ts`). ZSET in Redis-likes; secondary index on `ts` in DynamoDB; etc.

### Per-call constraints

- Reads MUST return copies. Mutating a returned profile or payload MUST NOT affect storage.
- All operations are async. Synchronous backends should still wrap return values in `Promise.resolve(...)`.

## Coding standards

- TypeScript strict mode.
- ESM-first (CJS interop is generated by `tsup`).
- No runtime dependencies in `dependencies`. Backend SDKs go in `peerDependencies`.
- 70%+ line/function coverage on `src/**/*.ts`.

## Pull requests

1. Fork + branch from `main`.
2. Open a PR with a clear description and (if applicable) a conformance suite report.
3. CI will run typecheck + the in-memory suite. If you've added an adapter that requires live infrastructure, gate its tests on an env var (see `tests/redis.test.ts` and `tests/upstash.test.ts` for examples).

## Code of Conduct

We follow the [Contributor Covenant 2.1](CODE_OF_CONDUCT.md). Be kind, be specific, assume good faith.
